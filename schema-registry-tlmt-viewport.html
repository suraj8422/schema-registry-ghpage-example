PageView

pageView enables us to know 'how many times was X screen viewed?''

-- Firing --
A pageView event fires when a screen immediately starts loading. See consult with gkagan & tabend in https://github.bamtech.co/schema-registry/schema-registry/commit/dda368bce91fd78e2956d98010cd084a772a5a3a .
A pageView event fires also when changing orientation updates layout. Because the entire Page has updated, the business wants to track this explicitly.
A pageView event fires when a user returns to a screen they were previously on (i.e.back nav or returning to 'Home').

-- Examples --
- if a user is on the home page, clicks on the pixar collection, clicks 'Up' and then clicks the back button to go back to the pixar page, 4 pageView events
would fire with a different pageView.
- if a user is on the home page, clicks on the pixar collection, clicks 'Up' and then clicks the back button to go back to the pixar page, 4 pageView events
would fire with a different pageView.
- if a user is on the home page, clicks on the pixar collection, clicks 'Up' and then clicks on the Home menu_item to go back to the Home page, 4 pageView events
would fire with a different pageView.
Not a pageView e.g. - scrolling up/down to load more carousels
Not a pageView e.g. - scrolling left/right thru a carousel to load more tiles

Input

input allows us to know whenever a user inputs anything into the app. An input event is fired once the user stops typing for a certain amount of time (200 milliseconds) or changes the state of checkbox or toggle. After the user has stopped typing for 200ms, the event should fire with whatever is in the search input_form element. Examples of this are typing in the Search Container or if the user is updating their Profile Name. -- Examples -- - user types 'Frozen' into the search browser, waits 201 ms, 1 event should fire with the entire string 'Frozen'. - if a user types 'Sta', waits 199ms, then types 'r', and waits 201ms, 1 event will fire with 'Star'. If the user then deletes 'r' and waits 201ms, 1 event should fire with the string 'Sta'.

ContainerView

containerView enables us to answer 'Which containers were viewed on a page?' It is captured when any part of the Container is presented in the UI (regardless of how quickly it was actually viewed). A containerView event should fire when any part of an element (>= 1 pixel up to 100% present) is on the page (even if it's not clickable). New containerView events should fire when a user returns to a screen they were previously on (i.e. back nav or returning to 'Home') and re-impresses the containers on that page. containerView should include an array elements which gives details on the current Elements being shown. This should include both Elements that are only peeking (1 pixel) as well as Elements that are 100% in view This will enables us to identify 'What exactly the user saw within this container?' -- Horizontal scroll -- The containerView Events will only fire the first time the element comes into view on that page. If an element is only peeking, and then fully comes into view when the user scrolls horizontally, a container view event will only fire once for that element (the first it was viewed.) Example of this below. If a user is scrolling quickly through a container, a containerView event should fire once the user stops scrolling and should contain all elements that were scrolled by. -- Vertical Scroll -- When a new container comes into view an event should only be fired when the container first comes into view (peeking or fully in view). If a user is scrolling quickly, a containerView event should fire for each container that is scrolled by even if it was only in view for a small amount of time. -- Hero Carousel Activity-- The hero container should only fire when the hero is in the user's viewport and a new element is being shown to the user. The hero container should fire the first time an element comes into view on that pageView instance. If the user has already seen the same hero tile during the pageView instance, we should not receive another containerView for that tile. (i.e. there should be no re-impressions, so each tile in the hero will only fire once for that pageView instance) -- Examples -- - Search results that result in new containers and elements shown, should fire a new containerView event. If the containers and elements change for each letter typed in the search bar, we expect a containerView event to be fired each time. (i.e user types 'S' and pauses for >200ms, a containerView event fires with containers and elements that are visible at that time, user then types 't' and pauses for >200ms, another containerView event fires with new elements that are visible). - If a user scrolls quickly down a page, events should fire for each container that is viewed (or scrolled past), regardless of the speed of the scroll. - If the hero is in the user's viewport and has a photo of the Mandalorian in full view (elementIndex = 0), one event should fire including the Mandalorian encodedSeriesId. Once the Hero rotates and the next piece of content is in view (elementIndex = 1), a second event should fire (assuming the hero is still in the user's viewport). - If the user scrolls down the page so that the hero is no longer in the viewport and the hero continues to rotate (past elementIndex = 2, 3, 4), we should NOT receive new containerView events for the hero. We should only receive hero containerView events when the hero is in the user's viewport - If the user scrolls back up to the top of the Home page so the hero is now back in view and focused on elementIndex = 5, we should receive one containerView for the tile at elementIndex = 5 - Once the user has seen all hero tiles in their viewport (i.e. 10 hero containerViews), we should stop receiving hero carousel containerView events for this pageView instance (since these would be re-impressions). - For Content Containers, if the user can view container 1, which has 4 elements fully in view and 1 element partially in view, one event will fire with 5 elements in the payload. Once the user scrolls to the right , a second container view event will fire that includes any new elements in view (6,7, etc.). - On BBD's or connected devices, where a user needs to focus on a specific element, if a user is focused on container 1, element 3, one event will fire with all elements in view (1,2,3,4,5). Once the user scrolls one tile to the right and is now focused on element 4, a second event will fire with all elements in view (2,3,4,5,6). - the search bar should have it's own container as well as the cta buttons on the movie details pages and the suggested/extras/detail menu_items. This will be mocked up in JIRA ticket attachments. - the Video Player itself and Up Next should each have its own containers.
